Business Components (BC)

The foundation of Siebel data model

A Business Component is the basic unit of the Siebel data model. It maps to a database table and provides a logical interface for working with data.

üí° Key Properties

  • Name: Unique identifier (e.g., "Account", "Contact")
  • Table Name: Physical database table (e.g., "S_ORG_ORG")
  • Fields: Columns mapped to the table
  • Primary Key: Unique identifier (usually ROW_ID)
  • Parent BC: Hierarchical relationships
  • Audit Fields: CREATED, MODIFIED, CREATED_BY, MODIFIED_BY

Business Objects (BO)

Organize and relate multiple Business Components

A Business Object is a logical grouping of related Business Components. It defines parent-child relationships and coordinates data access across multiple tables.

üí° Key Attributes

  • Name: Logical name (e.g., "Account")
  • Primary BC: Main component (e.g., "Account BC")
  • Child BCs: Related components (e.g., "Contact BC")
  • Relationships: Foreign keys linking BCs
  • Properties: Default sorting, access control

UI components that display and edit data

Joins

Connect data from multiple tables

Joins link multiple Business Components together, enabling data from related tables to be displayed in the same applet. They define the relationship between tables.

üí° Join Properties

  • Primary BC: Main data source
  • Foreign BC: Related data source
  • Join Type: Inner Join, Left Outer Join
  • Foreign Key: Linking field (e.g., ACCOUNT_ID)
  • Cardinality: 1-to-1 or 1-to-many relationship

üîÑ Join Types

INNER JOIN       ‚Üí Only matching records from both tables
LEFT_OUTER_JOIN  ‚Üí All primary records + matching foreign records
RIGHT_OUTER_JOIN ‚Üí All foreign records + matching primary records
FULL_OUTER_JOIN  ‚Üí All records from both tables

    Field Properties & Data Types

    Search Specifications (Query Language)

    Search Specifications define how to query data from Business Components. They use a SQL-like query language.

    Basic Search Syntax

    [Field] = "Value"              - Exact match
[Field] LIKE "%Value%"         - Contains
[Field] > 1000                 - Greater than
[Field] < 1000                 - Less than
[Field] >= 1000                - Greater or equal
[Field] <= 1000                - Less or equal
[Field] != "Value"             - Not equal
[Field] IN ("A", "B", "C")     - Multiple values
[Field] IS NULL                - Empty field
[Field] IS NOT NULL            - Not empty

    Complex Search Examples

    Example 1: Find active accounts with revenue > 1M
[Status] = "Active" AND [Revenue] > 1000000

Example 2: Contacts matching multiple names
[FirstName] LIKE "%John%" OR [FirstName] LIKE "%Jane%"

Example 3: Orders from last 30 days
[CreatedDate] >= TRUNC(SYSDATE) - 30

Example 4: Accounts without owner
[OwnerId] IS NULL

Example 5: Complex business logic
([Status] = "Active" AND [Revenue] > 500000) 
OR 
([Industry] = "Technology" AND [Region] = "North America")

    Search Best Practices

    ‚úÖ Index frequently searched fields
‚úÖ Use exact match instead of LIKE when possible
‚úÖ Avoid OR conditions (use IN instead)
‚úÖ Test queries with large datasets
‚úÖ Use wildcards strategically (%Value%, not Value%)
‚úÖ Avoid searches on calculated fields
‚úÖ Use date ranges instead of date functions

    Indexes & Database Optimization

    Indexes improve query performance by allowing faster lookups, but they consume disk space and slow down inserts/updates.

    When to Create Indexes

    ‚úÖ ON: Fields used frequently in WHERE clauses
‚úÖ ON: Fields in JOIN conditions
‚úÖ ON: Fields in ORDER BY clauses
‚úÖ ON: Foreign key fields

‚ùå AVOID: Small tables (< 10K rows)
‚ùå AVOID: Fields with few unique values (Status)
‚ùå AVOID: Large text fields (CLOB, VARCHAR2(4000))
‚ùå AVOID: Fields modified frequently

    Index Types

    SINGLE COLUMN    - Index on one field
                   Example: [Name]

COMPOSITE INDEX  - Index on multiple fields
                   Example: [LastName, FirstName]
                   (Often called compound index)

UNIQUE INDEX     - Enforces uniqueness
                   Example: [Email] must be unique

FULL TEXT INDEX  - Searches text content
                   Example: [Description] LIKE "%keyword%"
